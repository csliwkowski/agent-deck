<objective>
Fix uppercase/shifted key shortcuts and text input on Wayland compositors by disabling the Kitty keyboard protocol at TUI startup and providing a CSI u input filter as fallback.

Purpose: On Wayland, modern terminals (Ghostty, Foot, Alacritty) send Kitty keyboard protocol sequences (CSI u format) by default. Bubble Tea v1.3.10 does not parse these sequences, causing them to be silently dropped. This means uppercase shortcuts (M, R, F, etc.) and uppercase text input are completely broken on Wayland.

Output: A keyboard compatibility layer that: (1) sends escape sequences to disable the Kitty protocol at TUI startup and restores it on exit, and (2) provides a CSI u input filter as fallback for terminals that might not fully honor the protocol disable.
</objective>

  <behavior>
    - ParseCSIu("\x1b[109;2u") returns KeyMsg with Runes=['M'] (Shift+m = uppercase M)
    - ParseCSIu("\x1b[114;2u") returns KeyMsg with Runes=['R'] (Shift+r = uppercase R)
    - ParseCSIu("\x1b[102;2u") returns KeyMsg with Runes=['F'] (Shift+f = uppercase F)
    - ParseCSIu("\x1b[109u") returns KeyMsg with Runes=['m'] (no modifier = lowercase m)
    - ParseCSIu("\x1b[97;5u") returns KeyMsg for Ctrl+a (modifier 5 = ctrl)
    - ParseCSIu("\x1b[13u") returns KeyEnter (codepoint 13 = CR)
    - ParseCSIu("not a csi u") returns nil (not a CSI u sequence)
    - CSIuReader wrapping a reader containing "\x1b[109;2u" produces bytes for "M" when read
    - CSIuReader passes through normal ASCII bytes unchanged
    - CSIuReader passes through standard escape sequences unchanged (e.g., "\x1b[A" for up arrow)
    - DisableKittyKeyboard writes "\x1b[>0u" to the provided writer
    - RestoreKittyKeyboard writes "\x1b[<u" to the provided writer
  </behavior>
  <action>
    Create `internal/ui/keyboard_compat.go` with:

    1. **DisableKittyKeyboard(w io.Writer)**: Write `\x1b[>0u` to push keyboard mode 0 (legacy) on the terminal. This tells Kitty-protocol-aware terminals to stop sending CSI u sequences and fall back to legacy key reporting. Safe to call on terminals that don't support the protocol (they ignore unknown escape sequences).

    2. **RestoreKittyKeyboard(w io.Writer)**: Write `\x1b[<u` to pop the keyboard mode stack, restoring the terminal's previous keyboard mode. Must be called on TUI exit.

    3. **ParseCSIu(data []byte) *tea.KeyMsg**: Parse a CSI u sequence (`ESC [ codepoint ; modifiers u`) into a tea.KeyMsg. The Kitty keyboard protocol encodes keys as `\x1b[<unicode_codepoint>;<1+modifier_bits>u` where modifier bits are: 1=shift, 2=alt, 4=ctrl. For shifted letters (modifier bit 1 set), the codepoint is always lowercase; apply shift to produce uppercase. For codepoints that map to control keys (13=Enter, 9=Tab, 27=Esc, 127=Backspace), return the appropriate tea.KeyType. Return nil if the sequence is not a valid CSI u format.

    4. **NewCSIuReader(r io.Reader) io.Reader**: A wrapper reader that intercepts CSI u sequences in the byte stream and translates them into legacy byte sequences that Bubble Tea can parse. This is a fallback for terminals that might not honor the protocol disable. Implementation: buffer incoming bytes, scan for `\x1b[<digits>;<digits>u` patterns, replace with the equivalent legacy bytes, pass everything else through unchanged. Use a small internal buffer (256 bytes) to avoid allocations.

    IMPORTANT: The CSI u regex pattern is `\x1b\[(\d+)(;\d+)*u`. The codepoint is the first number group, modifier is the optional second group (default 1 = no modifiers). Modifier value is 1 + bitmask (shift=1, alt=2, ctrl=4), so modifier=2 means shift only.

    For special codepoints: 13 -> KeyEnter, 9 -> KeyTab, 27 -> KeyEscape, 127 -> KeyBackspace, 32 -> KeySpace. For regular codepoints (letters, numbers, symbols), emit the rune. If shift modifier is set and the codepoint is a lowercase letter (97-122), emit the uppercase letter (65-90).

  <name>Task 2: Integrate keyboard compatibility into TUI startup</name>
  <files>cmd/agent-deck/main.go</files>
  <action>
    In `cmd/agent-deck/main.go`, add the Kitty keyboard protocol disable/restore around the TUI program:

    1. **Before `tea.NewProgram`** (around line 468), call `ui.DisableKittyKeyboard(os.Stdout)` to push legacy keyboard mode. This MUST happen after any other stdout writes (like the web server address) but before the Bubble Tea program starts.

    2. **After `p.Run()` returns** (around line 484), call `ui.RestoreKittyKeyboard(os.Stdout)` using a deferred call placed right after DisableKittyKeyboard. Use defer to ensure restoration happens even on error paths.

    3. **Add CSI u input filter as fallback**: Wrap stdin with `ui.NewCSIuReader(os.Stdin)` and pass it to `tea.NewProgram` via `tea.WithInput()`. This handles the edge case where a terminal sends CSI u sequences despite the protocol disable (e.g., during the brief window before the disable takes effect, or if the terminal ignores the disable).

    The modified tea.NewProgram call should look like:
    ```go
    ui.DisableKittyKeyboard(os.Stdout)
    defer ui.RestoreKittyKeyboard(os.Stdout)

    p := tea.NewProgram(
        homeModel,
        tea.WithAltScreen(),
        tea.WithMouseCellMotion(),
        tea.WithInput(ui.NewCSIuReader(os.Stdin)),
    )
    ```

    IMPORTANT: Using `tea.WithInput()` changes the input type from TTY to custom, which might affect raw mode setup. Check if Bubble Tea still enters raw mode when using WithInput. If not, we need to use `tea.WithInputTTY()` instead and find another way to inject the filter. Test by verifying that regular lowercase keys still work after the change.

    Actually, looking at the bubbletea source: when `WithInput` is used with a custom reader, bubbletea sets `inputType = customInput` and does NOT set up raw mode on the terminal. This would break all input. Instead, we should NOT use `tea.WithInput`. The DisableKittyKeyboard approach alone is sufficient because:
    - The escape sequence disables CSI u BEFORE bubbletea starts reading
    - Bubbletea then reads legacy sequences from the TTY
    - No input filter needed at all

    So the final approach is simpler:
    ```go
    ui.DisableKittyKeyboard(os.Stdout)
    defer ui.RestoreKittyKeyboard(os.Stdout)

    p := tea.NewProgram(
        homeModel,
        tea.WithAltScreen(),
        tea.WithMouseCellMotion(),
    )
    ```

    Keep the CSI u parser code in keyboard_compat.go as a public API for future use (e.g., if we need to handle CSI u sequences for other reasons), but don't wire up the input filter in main.go. The escape sequence approach is cleaner and more robust.
